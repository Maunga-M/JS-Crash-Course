Comparison Operators

Comparison operators are used to make decisions in your code by checking how two values relate to each other. A comparison operator takes two values, compares them, and produces a boolean result. There are eight comparison operators in JavaScript. Click each card to see what it does.

Comparing numbers

You may recognise the first four operators from your early math classes in school and in JavaScript they work exactly as you would expect with numbers.

JavaScript 
console.log(10 > 3)   // true
console.log(3 < 1)     // false
console.log(10 >= 10)  // true
console.log(12 <= 10)  // false
console.log(5 > 10)   // false
Output

Comparing strings

Comparison of strings are a not so obvious. JavaScript compares strings lexicographically, which means character by character, using each character's Unicode value. This is similar to alphabetical order, but with one important difference: uppercase letters come before lowercase ones in Unicode. So "B" is considered less than "a".

JavaScript 
console.log("apple" < "banana")  // true - "a" comes before "b"
console.log("cat" > "car")       // true - "t" comes after "r"
console.log("apple" < "Zebra")   // false - all uppercase come before any lowercase
console.log("DOG" > "ELEPHANT")   // false - 'D' comes before 'E'
Output

The string number trap

When comparing two strings that look like numbers, JavaScript does not convert them to numbers first. It compares them as text, character by character. This leads to a result that surprises almost every beginner.

JavaScript 
// Comparing numbers: 10 is greater than 2
console.log(10 > 2)      // true

// Comparing strings: "1" comes before "2" in Unicode, so "10" < "2"
console.log("10" > "2")  // false
console.log("2" > "10")  // true
Output
Why this happens

JavaScript looks at the first character of each string. E.g "1" versus "2". Since "1" comes before "2" in Unicode, "10" is considered less than "2", no matter how many digits follow. If you need to compare numeric strings correctly, convert them to numbers first using Number() or parseInt().

Loose and Strict Equality

Loose equality == performs type coercion before comparing. Strict equality === does not. It checks value and data type simultaneously. In practice, always use strict ===. Loose equality produces results that are difficult to predict.

JavaScript 
console.log("5" == 5)   // true - coercion converts "5" to 5
console.log("5" === 5)  // false - different types
console.log(0 == false)   // true - 0 converts to false, false stays false
console.log(0 === false)  // false - number vs boolean
console.log("" == false) // true - "" converts to 0, false converts to 0
console.log("" === false) // false - different types
Output

Loose and Strict Inequality

The same distinction applies for inequality. Loose != coerces the data types and then comapres values. Strict !== does not. It checks both data type and value. Prefer !== for the same reasons.

JavaScript 
console.log("5" != 5)  // false - coercion makes them equal
console.log("5" !== 5)  // true - different types
console.log(1 != 2)    // true - different values
console.log(1 !== 2)   // true - different values, same type

Logical Operators

Comparison operators evaluate one condition. Logical operators let you combine or invert conditions so you can make more complex decisions in your code. They evaluate expressions and return a result based on how those conditions relate to each other. We have three logical operators.

&&   AND
Evaluates two conditions. Returns a truthy result only if both sides are truthy.
age > 18 && hasLicence
||   OR
Evaluates two conditions. Returns a truthy result if at least one side is truthy.
isStudent || isStaff
!   NOT
Inverts a boolean value. Turns true into false and false into true.
!isExpired

Core Concept

Operators in Action

In code, A and B in the truth table are usually comparison expressions
e.g (isAdmin === false) && (age === 19).
The logical operator evaluates each side and combines their results. All the results below are booleans because the operands (A and B) are comparisons and boolean variables.
Note: An operand is a value that an operator acts on.

JavaScript 
let age = 22
let hasLicence = true
let isStudent = false

console.log(age > 18 && hasLicence)  // true -> both truthy
console.log(age > 18 && isStudent)   // false -> isStudent is false
console.log(age > 18 || isStudent)   // true -> age > 18 is truthy
console.log(!hasLicence)              // false - inverts true
console.log(!isStudent)               // true - inverts false
Output

Core Concept

Non-Boolean Results

You might expect && and || to always return true or false. They do not. They return one of the actual values you gave them, depending on which one determined the outcome.

With &&, JavaScript evaluates left to right and returns the first falsy value it finds because as soon as one side is falsy, the whole expression cannot be true. If every value is truthy, there is nothing to fail on, so it returns the last value.

With ||, JavaScript evaluates left to right returns the first truthy value it finds because as soon as one side is truthy, the whole expression is satisfied. If every value is falsy, there is nothing to return as a success, so it returns the last value.

In both cases, JavaScript stops the moment it has enough information to decide. This is called short-circuiting. The returned value is whatever caused JavaScript to stop.

JavaScript 
// && returns the first falsy value, or the last value if all are truthy
console.log(true && "hello")  // "hello" -> left is truthy, returns right
console.log(false && "hello") // false -> left is falsy, returns left
console.log(0 && "hello")     // 0 -> 0 is falsy, returns left

// || returns the first truthy value, or the last value if all are falsy
console.log(false || "hello") // "hello" -> left is falsy, returns right
console.log("hi" || "hello")   // "hi" -> left is truthy, returns left
console.log(0 || 42)           // 42 -> left is falsy, returns right
Output

Here is another example to help you understand this behaviour

JavaScript 
let count = 0

// Left is false, so right side never runs
false && (count += 1)
console.log(count)  // 0 -> right side was skipped

// Left is true, so right side never runs
true || (count += 1)
console.log(count)  // 0 -> right side was skipped again

// Left is false, so right side DOES run
false || (count += 1)
console.log(count)  // 1 -> right side ran this time
Output
This is not a bug

This behaviour is intentional and used constantly in JavaScript. Short-circuiting is also used as a concise way to provide a fallback value when a variable might be empty or undefined. See the example below.

JavaScript 
let username = ""
let display = username || "Guest"
console.log(display)  // "Guest" -> username is falsy so || returns right side

let loggedIn = "Priya"
let display2 = loggedIn || "Guest"
console.log(display2)  // "Priya" -> loggedIn is truthy so || returns left side
Output

Core Concept

Operator Precedence

When multiple operators appear in the same expression, JavaScript evaluates them in a fixed order. Comparison operators are evaluated before logical operators. Among the logical operators, ! is evaluated first, then &&, then ||. Parentheses override everything.

JavaScript 
console.log(5 > 3 || 10 < 2 && 4 === 4)
// Comparisons run first, then &&, then ||
// 5 > 3 || 10 < 2 && 4 === 4
// true || false && true -> && is evaluated first 
// true || false
// true
Output
JavaScript 
console.log((5 > 3 || 10 < 2) && 4 === 4)
// Parentheses force a different grouping
// (5 > 3 || 10 < 2) && 4 === 4
// (true || false) && true -> parentheses are evaluated first
// true && true
// true
Output
JavaScript 
// ! only flips the value immediately to its right
console.log(!true && false)   // false -> (!true) is false, false && false is false

// Parentheses change what ! operates on
console.log(!(true && false))  // true -> (true && false) is false, !false is true

// Without parentheses, ! runs first before && or ||
console.log(!true || true)    // true -> (!true) is false, false || true is true
Output
When in doubt, use parentheses

Relying on precedence rules in complex expressions makes code harder to read. If you have more than two conditions combined, group them explicitly with parentheses. It makes your intention clear and removes any ambiguity.

Summary

Module Summary

Key takeaways

  • There are eight comparison operators. Always prefer === and !== over == and != to avoid coercion surprises.
  • Strings are compared lexicographically, character by character. "2" is greater than "10" because "2" comes after "1" in Unicode.
  • && requires both sides to be truthy. || requires at least one. ! inverts a boolean.
  • && returns the first falsy value, or the last value if all are truthy. || returns the first truthy value, or the last value if all are falsy.
  • Evaluation order: comparisons first, then !, then &&, then ||. Use parentheses to make complex expressions explicit.
  • Short-circuiting means JavaScript stops evaluating as soon as the result is determined.
